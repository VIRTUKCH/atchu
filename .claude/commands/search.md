# 웹 검색 스킬

사용자가 `/search <질문>` 으로 호출한다.
검색 질문: $ARGUMENTS

## 실행 순서

아래 단계를 **반드시 순서대로** 실행한다.
단계를 건너뛰거나 합치지 않는다.

---

### Step 1. 의도 파악

사용자의 질문을 분석해서 아래 항목을 정리한다.

```
[의도 파악]
- 핵심 질문: (한 줄 요약)
- 검색 목적: (학습 / 비교 / 문제 해결 / 최신 정보 확인)
- 필요한 정보 유형: (공식 문서 / 블로그 / 논문 / 뉴스 / 커뮤니티)
```

---

### Step 2. 검색 계획 제안

검색을 실행하기 **전에** 아래 계획을 사용자에게 보여준다.

```
[검색 계획]
- 쿼리 1: "..."
  - 대상 도메인: [example.com, ...]
  - 선정 이유: ...
- 쿼리 2: "..." (필요시)
  - 대상 도메인: [...]
  - 선정 이유: ...
- 검색 기간: (전체 / 최근 1년 / 최근 1개월 / 최근 1주)
  - 기간 선정 이유: ...
- 검색 언어: (한국어 / 영어 / 둘 다)
```

**도메인 선정 가이드:**
- 개발/기술: `stackoverflow.com`, `github.com`, 공식 docs 도메인
- 한국어 개발: `velog.io`, `tistory.com`
- 뉴스/시사: `reuters.com`, `bloomberg.com`
- 학술: `arxiv.org`, `scholar.google.com`
- 목적에 맞는 도메인이 명확하지 않으면 도메인 필터 없이 검색한다.

---

### Step 3. 사용자 확인 (필수)

Step 2의 계획을 보여준 뒤, **반드시 사용자에게 확인을 받는다.**
아래와 같이 질문한다:

> 이대로 검색할까? 수정하고 싶은 거 있으면 말해줘.
> - 쿼리 변경
> - 도메인 추가/제거
> - 기간 변경
> - 언어 변경

사용자가 "ㅇㅇ", "고", "진행해" 등 승인하면 Step 4로 넘어간다.
수정 요청이 있으면 Step 2를 수정해서 다시 보여준다.

**중요: 사용자 확인 없이 절대 검색을 실행하지 않는다.**

---

### Step 4. 검색 실행

Step 2에서 확정된 계획대로 `WebSearch`를 실행한다.

**병렬 검색 원칙:**
- 설계한 쿼리가 여러 개면 **모든 WebSearch를 동시에 병렬 호출**한다. 순차 실행 금지.
- 검색 결과에서 유용한 페이지가 여러 개면 **모든 WebFetch도 동시에 병렬 호출**한다.
- WebFetch 대상은 아끼지 말고, 관련성 있어 보이는 페이지는 **최대한 많이** 가져온다.

**검색 규칙:**
- `allowed_domains`에 확정된 도메인을 넣는다.
- 검색 기간이 지정되어 있으면 쿼리에 연도/기간을 포함한다.
- 검색 결과가 부족하면 도메인 필터를 풀고 재검색한다.

---

### Step 5. 결과 정리

```
[검색 결과]
- 답변: (핵심 내용 요약)
- 신뢰도: (높음 / 보통 / 낮음) + 판단 근거
- 출처:
  1. [제목](URL) — 핵심 내용 한 줄
  2. [제목](URL) — 핵심 내용 한 줄
```

---

## 디버깅 규칙

- 사용자가 결과에 불만이면, **어느 단계가 문제인지** 먼저 파악한다.
  - 의도 파악이 틀렸으면 → Step 1부터 다시
  - 쿼리/도메인/기간이 안 맞았으면 → Step 2부터 다시 (Step 3 확인도 다시)
  - 결과 해석이 잘못됐으면 → Step 5만 다시
- 재검색 시에도 동일한 포맷을 유지한다.
