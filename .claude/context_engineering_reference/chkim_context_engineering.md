# 1인 개발자의 컨텍스트 엔지니어링 적용기

## 나의 상황

- 1인 기획자 + 프론트엔드/백엔드 개발자
- PR 없음, 코드 리뷰 없음, 팀원 없음
- 기획부터 배포까지 혼자 하고, AI(Claude)가 유일한 협업 상대

팀에서 쓰는 것들 — 멀티 에이전트 역할 분리(TPM AI / MSA AI), PR 워크플로우, JSON DSL, 팀 그라운드 룰 — 은 다 필요 없다. 나한테 필요한 건 딱 하나: **클로드가 깨어났을 때, 내 머릿속에 있는 기획 의도와 프로젝트 맥락을 최소한의 토큰으로 정확히 전달하는 것.**

---

## 설계 원칙

### 1. 기획 → 설계 → 코드, 단방향 흐름

```
README.md (왜 만드는가)
  → .claude/docs/pages/*.md (각 페이지를 어떻게 만드는가)
    → src/ (실제 코드)
```

- `README.md`는 제품 전략의 원천. 서비스 정의, 페르소나, 가치 제안, 설득 방법이 여기에 있다.
- `.claude/docs/pages/`는 페이지별 구현 설계서. 타겟 사용자, 유입 경로, 보여줄 것, CTA가 적혀 있다.
- 클로드가 랜딩 페이지를 고칠 때, README.md에서 "왜"를 읽고 landing.md에서 "어떻게"를 읽은 다음 코드를 건드린다.

기획 의도를 모르는 채 코드만 고치면 방향이 틀어진다. 이 흐름이 그걸 막는다.

### 2. CLAUDE.md = 진입점

클로드가 세션을 시작하면 제일 먼저 읽는 파일. 여기에는:

- 프로젝트가 뭔지 (한 줄)
- 필수 명령어 (빌드, 파이프라인)
- 워크플로우 규칙 (README 참고, CHANGELOG 기록, 커밋 전 문서 최신화)
- 반드시 물어볼 것 목록 (Human-in-the-Loop)
- 자동 커밋 규칙
- 디렉터리 구조 전체 맵

모든 상세 문서는 CLAUDE.md에서 링크로 참조한다. CLAUDE.md 자체는 64줄로 짧게 유지한다. 여기서 전체 구조를 파악하고, 필요한 문서만 선택적으로 로드하는 구조.

### 3. 파일 단위 분리 = 토큰 절약

컨텍스트 윈도우는 유한하다. 모든 정보를 하나의 거대한 문서에 넣으면 토큰을 낭비한다.

```
.claude/docs/
  pages/          — 페이지별 기획 (7개 파일)
  law_risk/       — 법적 리스크
  marketing/      — 마케팅 전략
  development/    — 데이터 파이프라인
  tax/            — 세금/사업자
  operations/     — 운영/이용약관
```

랜딩 페이지를 고칠 때 세금 문서를 읽을 필요가 없다. 파이프라인을 수정할 때 마케팅 문서를 읽을 필요가 없다. 관심사별로 파일을 나눠놓으면 클로드가 필요한 것만 읽는다.

### 4. Human-in-the-Loop 설계

CLAUDE.md에 "반드시 사용자에게 물어볼 것" 목록을 명시했다:

- 새로운 기능/기획 방향 결정
- 아키텍처 변경 (새 라이브러리, 디렉터리 구조)
- docker, 배포 등 외부 서비스 실행
- .env 또는 시크릿 관련 변경
- 파괴적 git 작업
- 에러 3회 이상 반복 시

반대로, 이 목록에 없는 것(파일 CRUD, npm, 일반 git 작업)은 물어보지 않고 자동으로 한다. 1인 개발자는 사이드 프로젝트에 최소한으로 개입하고 싶기 때문에, 물어볼 것과 자동으로 할 것의 경계를 명확히 그었다.

### 5. 자동 커밋 = 롤백 포인트 확보

```
- 하나의 기능/버그 수정이 완료되면 자동으로 커밋 (사용자에게 묻지 않음)
- 큰 변경 전에도 커밋하여 롤백 포인트 확보
```

혼자 작업하니까 PR도 없고 리뷰도 없다. 대신 자주 커밋해서 언제든 되돌릴 수 있게 했다. 이건 AI에게 자율성을 주되, 되돌릴 수 없는 상황은 막는 안전장치다.

## 팀에서 쓰는 것 vs 내가 쓰는 것

| 팀에서 쓰는 것 | 내가 쓰는 것 | 이유 |
|---|---|---|
| TPM AI / MSA AI 역할 분리 | 클로드 하나 | 1인이라 관심사 분리가 필요 없음 |
| JSON DSL로 토큰 최적화 | 마크다운 자연어 | 문서를 사람도 읽어야 하므로 가독성 우선 |
| 복잡한 스킬 시스템 | `/search` 하나 | 웹 검색 외에 자동화할 반복 작업이 없음 |
| PR 워크플로우 | 자동 커밋 | 리뷰어가 없으므로 롤백 가능성만 확보 |
| 팀 그라운드 룰 | CLAUDE.md 워크플로우 | 합의 대상이 나 혼자이므로 규칙만 적으면 됨 |
| Layered Architecture (Global/Domain/Local) | 단일 레포 `.claude/docs/` | 전사 공통 레이어가 존재하지 않음 |

## 실제 파일 구조와 역할

```
README.md                          ← 기획의 원천 (why)
CLAUDE.md                          ← 클로드 진입점 (64줄, 전체 맵)
.claude/
  settings.local.json              ← 권한, hooks, 모델 설정
  commands/search.md               ← 유일한 커스텀 스킬
  docs/
    pages/                         ← 페이지별 구현 설계 (what/how)
      landing.md
      trend_list.md
      trend_detail.md
      market_overview.md
      faq.md
      columns.md
      discord_alert.md
    law_risk/                      ← 법적 리스크
    marketing/                     ← 마케팅 전략
    development/                   ← 데이터 파이프라인
    tax/                           ← 세금/사업자
    operations/                    ← 운영
    solo_developer_checklist.md    ← 6개 영역 종합 인덱스
    CHANGELOG.md                   ← 작업 기록
  context_engineering_reference/   ← 이 문서가 있는 곳
```

## 요약

컨텍스트 엔지니어링의 핵심은 "LLM에게 올바른 정보를 올바른 타이밍에 주는 것"이다. 1인 개발자에게 이것은:

1. **기획 의도를 문서로 남긴다** — 내 머릿속에만 있으면 클로드가 모른다
2. **관심사별로 파일을 나눈다** — 토큰은 유한하므로 필요한 것만 로드한다
3. **진입점(CLAUDE.md)에서 전체 맵을 제공한다** — 클로드가 어디를 봐야 할지 안다
4. **자율성의 경계를 명시한다** — 물어볼 것과 알아서 할 것을 구분한다
5. **문서와 코드를 동기화한다** — 커밋 전에 문서를 최신화하는 규칙

팀 도구(멀티 에이전트, 스킬 체계, PR 워크플로우)는 팀이 필요해서 쓰는 거지, 컨텍스트 엔지니어링의 본질이 아니다. 본질은 "AI가 내 의도를 정확히 이해하도록 정보를 구조화하는 것"이고, 이 프로젝트는 그걸 1인 개발자 규모에서 실천한 사례다.
